# 技術寫作中的研究：業界做法與寫作系統整合分析

## 執行摘要

本研究探討技術寫作者在創作過程中如何進行線上研究，以及這些研究成果如何被組織和使用。研究目的是了解業界的研究實踐，作為未來將研究功能整合到我們寫作系統的基礎。

經過調查，我們發現幾個關鍵洞察：

- **研究是迭代的，不是單一前期階段。** 業界共識是研究至少發生在三個時間點：寫作前的主題探索、起草中的知識補充、以及編輯時的事實查核。
- **研究有明確的類型分野。** 不同類型的研究服務不同目的——背景研究建立脈絡、競品分析找到差異化角度、事實查核確保準確性——混為一談會導致效率低落。
- **研究成果需要獨立的文件形式。** 業界普遍使用「研究筆記」作為從研究到寫作的橋接文件，包含帶來源引用的摘要、關鍵發現、以及如何在文章中使用的建議。
- **我們的系統目前缺乏研究環節。** 現有流程直接從「作者已有的材料」出發，沒有支援作者在寫作前或寫作中進行系統性的外部研究。

## 背景與脈絡

我們的寫作系統目前由三個技能組成一條流水線：寫作管理（workspace 初始化、想法池）→ 文章準備（brief 完成、訪談蒐集材料、建立大綱）→ 文章寫作（起草、審查、修訂）。系統的核心設計原則是「材料神聖」——文章內容只來自作者提供的素材，絕不捏造。

這個設計在作者已經擁有充足經驗和素材時運作良好。但在實際寫作場景中，作者經常需要做額外的研究來補充自己的知識：查找最新的技術趨勢、確認某個 API 的行為、了解社群對某個做法的看法、或是找到支持論點的數據。目前系統沒有為這類需求提供結構化的支援。

## 研究問題與發現過程

使用者的初始問題是：「技術寫作者一般怎麼在網路上做研究？有哪些不同的類型或階段？什麼時候做？這些做法有什麼特性可以加入我們的寫作系統？」

經過釐清，我們確認了研究方向：

1. 先了解業界做法（本研究的主要範圍）
2. 研究涵蓋寫作前和寫作中兩種情境
3. 研究成果應產出獨立文件，附上參考連結供作者閱讀，並建議如何在文章中引用
4. 作者閱讀並確認研究成果後，這些成果自然成為材料的一部分

## 技術分析：業界研究實踐

### 研究的六種類型

業界將技術寫作中的研究分為幾種明確的類型，每種服務不同的目的：

**背景研究（Background Research）** 是最基礎的類型，目的是建立對主題的廣泛理解。這包含「初步研究」——在較廣泛的主題領域探索以找到具體的寫作角度，以及「深度研究」——針對已確定的題目蒐集具體內容素材。例如，一個作者打算寫關於 container orchestration 的文章，背景研究會幫他了解目前有哪些方案、各自的定位、以及社群的討論焦點。

**受眾研究（Audience Research）** 確認讀者是誰、他們的技術程度、以及他們搜尋時真正想找的是什麼（搜尋意圖）。這類研究通常透過分析搜尋關鍵字、檢視論壇討論中的常見問題、以及觀察讀者在類似文章下的回饋來進行。

**競品分析（Competitive Analysis）** 檢視同主題的現有文章，找出它們的缺漏和不足，以創造差異化價值。具體做法是搜尋目標關鍵字、閱讀搜尋結果第一頁的文章、記錄它們涵蓋和遺漏的重點。這類研究幫助作者回答「我的文章為什麼值得存在」。

**專家諮詢（SME Research）** 是與主題專家進行訪談或諮詢。技術寫作中，這可能是與開發某個功能的工程師對話、請教資深架構師的經驗、或是在社群中向專家提問。我們系統中的「訪談」步驟已經涵蓋了作者自身經驗的提取，但尚未支援外部專家的知識蒐集。

**事實查核（Fact-Checking）** 在技術寫作中特別重要，包含：測試程式碼範例確保可以運行、按照撰寫的步驟實際重現結果、使用多個來源交叉驗證事實、確認使用的是最新版本的文件。業界強調「絕不發布未經驗證的資訊」。

**數據與統計蒐集** 從產業報告、學術論文、官方文件中尋找支持論點的數據。例如，引用 Stack Overflow 調查的採用率數據、或是效能基準測試的結果。

### 研究的時間點：不是一個階段，而是三個

業界的共識是研究不只發生在寫作之前，而是貫穿整個寫作過程。具體來說，研究在三個時間點扮演不同角色：

**寫作前的主題研究** 是最直覺的階段。作者在確定題目後、開始起草前，進行背景研究、受眾研究和競品分析。目的是建立足夠的知識基礎來規劃文章結構，並找到差異化的角度。這個階段的產出通常是一份「內容簡報」（Content Brief），包含目標受眾、大綱、來源清單和寫作角度。

**起草中的補充研究** 發生在寫作過程中遇到知識缺口時。LibreTexts 的技術寫作教材指出：「寫作者很可能會在某些地方卡住，這表示需要停下來做更多研究或諮詢專家。」這類研究通常是針對性的——查找某個 API 的具體行為、確認某個配置選項的效果、或是尋找一個具體的數據來支持剛寫下的論點。

**編輯時的事實查核** 是最後一道品質關卡。在文章基本完成後，回頭驗證所有事實宣稱、測試所有程式碼範例、確認所有連結有效。這個階段的重點不是獲取新知識，而是確保已寫入的內容準確無誤。

值得注意的是，這三個階段不是嚴格線性的。寫作中的補充研究可能會改變大綱結構，事實查核可能會推翻某個論點需要重寫。業界將此描述為「迭代式流程」——當新資訊出現時，寫作者會回到先前的階段。

### 研究方法與技巧

技術寫作者在實際執行研究時，使用的方法可以歸納為幾大類：

- **搜尋與探索：** 使用 Google 進階搜尋運算子縮小結果、Google Scholar 搜尋學術資源、開發多組關鍵字列表、使用精確匹配搜尋
- **閱讀與消化：** 閱讀現有文章了解背景、檢視論壇討論（Stack Overflow、Reddit）了解常見問題、使用 Pocket/Instapaper 等工具收集來源
- **產品/技術測試：** 親自使用產品並測試功能、複製程式碼確認可運行、按照步驟重現結果
- **心智圖法：** 將主題放在中心繪製視覺化地圖，識別組成部分和知識缺口，防止研究偏離主題

業界的經驗法則是：每個主要論點大約需要 3-4 個高品質來源，一篇有四個要點的文章大約需要 12-16 個來源。

### 研究成果的組織方式

業界普遍的做法是建立帶有來源引用的結構化筆記。常見的格式包含：

**註釋式筆記**——每條摘要附上來源連結，區分個人改寫（消化後的理解）和直接引用（原文）。這種格式的好處是在寫作時可以快速判斷哪些內容需要引用來源。

**資源清單**——維護一份包含所有參考來源的單一文件，標記發布日期和可信度，定期檢視是否過時。

**從研究到大綱的轉化**——研究的最後一步是將有用的發現整理成大綱的素材。實務上的做法是注意研究中反覆出現的主題和趨勢，這些成為文章的主要論點。

### AI 輔助研究的現況

AI 工具正在改變技術寫作的研究流程。目前最被廣泛報導的實踐模式是「研究工具 + 寫作工具」的雙引擎模式：

使用 Perplexity AI 之類的工具進行帶有引用的深度研究（每個回應都附上行內引用連結到原始來源），然後將研究結果交給寫作工具進行結構化和起草。這個模式的核心價值是：AI 能快速彙整多個來源的資訊，但最終的判斷和篩選仍由作者完成。

根據 Nature Human Behaviour（2025）的研究，74% 的研究者已在工作流程中使用 AI 工具，61% 報告寫作和分析時間減少超過 40%。

## 我們系統的現況與缺口分析

將業界實踐與我們的系統對照，可以清楚看到幾個結構性的缺口：

**已涵蓋的部分：**
- 受眾研究：brief.md 的「Target Audience」欄位（who、background、prior state）已經涵蓋
- 專家諮詢（自身經驗）：訪談步驟有效地提取作者自身的經驗和知識
- 事實查核（部分）：寫作規則要求「使用作者的材料，絕不捏造」，但缺乏主動的外部事實查核機制

**缺口：**
- **背景研究：** 系統假設作者已經知道要寫什麼，沒有支援「探索主題、找到角度」的過程
- **競品分析：** 沒有機制讓作者了解同主題的現有文章，難以找到差異化角度
- **起草中的補充研究：** 寫作技能在遇到知識缺口時只能請作者補充，沒有系統性的外部研究支援
- **事實查核（主動）：** 審查迴圈檢查寫作規則違規，但不會主動查核文章中的事實宣稱
- **研究成果文件：** 沒有獨立的研究筆記格式，研究成果無處安放
- **來源追蹤：** 沒有機制記錄資訊來源和引用

## 解決方案探索：研究功能整合的可能方向

基於業界實踐和我們系統的設計原則，以下是幾個可能的整合方向：

### 方向一：獨立的「文章研究」步驟（寫作前）

在文章準備技能的訪談之前或之後，增加一個研究步驟。系統引導作者定義研究問題，然後透過 AI 輔助進行線上研究，產出一份獨立的研究筆記文件（例如 `research.md`），包含：帶來源連結的發現摘要、關鍵洞察、以及如何在文章中使用這些發現的建議。

- **優點：** 與現有流程的整合最自然，研究成果經作者確認後成為材料的一部分
- **考量：** 需要定義研究筆記的格式、決定研究範圍的控制機制

### 方向二：多時間點的研究支援

不只在寫作前，而是在三個時間點都提供研究支援：準備階段的主題研究、起草中的補充研究、以及審查時的事實查核。每個時間點的研究產出都附加到同一份研究文件中。

- **優點：** 更符合業界「研究是迭代的」共識
- **考量：** 複雜度較高，需要在三個技能中都加入研究相關的流程

### 方向三：研究作為獨立技能

將研究功能獨立為第四個技能（例如 `article-research`），可以在任何階段被呼叫。這個技能負責執行研究、組織成果、追蹤來源，產出的研究文件可以被其他技能引用。

- **優點：** 最大的靈活性，不侵入現有技能的流程
- **考量：** 需要定義技能之間的互動方式，可能需要修改 brief.md 格式來引用研究文件

### 各方向的共通需求

無論選擇哪個方向，都需要：

1. **研究筆記的文件格式：** 定義 `research.md` 的結構（研究問題、發現摘要、來源連結、使用建議）
2. **「材料神聖」原則的延伸：** 研究成果經作者閱讀確認後，成為合法的材料來源。未經作者確認的研究發現不能直接寫入文章。
3. **來源追蹤機制：** 在大綱的 Materials 中標記來源（author-sourced vs. research-sourced），確保可追溯性
4. **研究範圍控制：** 防止研究無限擴張，提供明確的「研究夠了」的判斷標準

## 建議與決策指引

基於分析結果，建議採用**方向一（獨立的文章研究步驟）作為起點**，並為未來擴展到方向二預留空間。

主要考量如下：方向一的複雜度最低，且最符合我們現有的「流水線」架構——在準備階段增加一個步驟比修改三個技能的風險小得多。同時，研究成果透過作者確認後成為材料的流程，完美延伸了「材料神聖」原則，不需要重新定義系統的核心設計。

具體來說，研究步驟可以插入在文章準備技能的「訪談」之前：先做研究建立主題的外部脈絡，再透過訪談提取作者的內部經驗，兩者結合形成更完整的材料基礎。

## 下一步行動計畫

實施需要分階段進行。第一階段重點是定義研究功能的具體規格，第二階段才是實作。

- **立即行動：** 如果決定推進，可以撰寫 PRD 定義研究步驟的詳細流程、研究筆記的文件格式、以及與現有技能的整合方式
- **需要決定的問題：**
  - 研究的觸發條件：每篇文章都做研究，還是作者可以選擇跳過？
  - 研究的深度控制：如何判斷「研究夠了」？
  - 研究文件的生命週期：是一次性的還是隨文章演進持續更新？
  - 研究類型的優先順序：先支援哪些類型的研究？（背景研究和競品分析可能是最高優先）
